# Accessibility Audit — 2026-04-19

> Phase A deliverable (accessibility arm) for the site-wide UX alignment initiative. See `docs/superpowers/plans/2026-04-19-ux-alignment.md`. The main inventory matrix lives at `docs/superpowers/audits/2026-04-19-ux-inventory.md`; this file deepens pattern #18 with WCAG 2.2 AA–focused findings citing real `file:line` references.

## Method

Source-only audit, no browser or assistive-tech walk-through. Scope: the five authenticated top-level routes (`/`, `/umonnun`, `/timar` incl. `/timar/reglulegir`, `/folk`, `/pappirar`) plus `/login`, shared primitives in `src/components/ui/`, navigation in `src/components/nav/` + `src/components/shared/`, feature folders in `src/components/{appointments,dashboard,info,log,recurringSeries,timar}`, and `src/app/[locale]/{(app),login}/**`.

Tools actually used:
- **Grep** across `src/` for icon-only buttons, `aria-label`, `sr-only`, tap-target class fingerprints (`h-8|h-9|h-10|h-11|size-icon*|min-h-9|min-h-11`), and color tokens.
- **Contrast math** computed locally in Python against the Bókasafn tokens defined in `src/app/globals.css` (paper `#faf7f0`, page `#f5f1e8`, ink `#2e342b`, ink-soft `#5a6158`, ink-faint `#8f958b`, sage `#8ba888`, sage-deep `#6b7f5c`, sage-shadow `#4f6243`, wheat `#c9b896`, wheat-deep `#a89770`, amber-bg-1 `#ede3c8`, amber-bg-2 `#e8d9b4`, amber-ink `#8a6b3a`, amber-ink-deep `#4a3a1f`, destructive `#b85a5a`). Alpha-compositing done per WCAG 2.x (25% sage over paper = `#dee3d6`, etc.) so the numbers below are the real on-screen ratios, not token-on-token.
- Checked the existing harness queue (`.claude/harness-improvements.md`) so already-logged issues are reconfirmed here with line references rather than re-opened.

Assumptions:
- Screen-reader semantics of Radix primitives (Dialog/Sheet/Tabs/Select/DropdownMenu/Switch/Label) are accepted as correct when the code uses them idiomatically; findings below flag only cases where the wrapping code breaks the primitive's guarantees.
- The project's **48 px tap-target floor** is stricter than WCAG 2.2's 24 px (2.5.8 minimum). Each tap-target finding lists both: the WCAG verdict (24 px) and the project verdict (48 px). A WCAG-PASS / project-FAIL entry still warrants a fix, but it's Major (not Blocker) under WCAG.
- The project typographic floor is **18 px body**. The shadcn defaults are `text-sm` (14 px) and `text-xs` (12 px), so any surface that uses those sizes for body content is flagged on the project floor even when WCAG 1.4.4 does not require a minimum size.

#### 1.3.1 Info and relationships (Level A)

- **`src/app/[locale]/(app)/page.tsx:70-87`** — Dashboard page renders a page-level eyebrow + an `<h2>` greeting but **no `<h1>`** at all. The only h1 in the authenticated shell is inside `src/app/[locale]/(app)/error.tsx:36` (reached only on error). Screen-reader users navigating by heading find h2 is the top of the page. - **`src/components/log/LogFeed.tsx:93-107`** — Author name is rendered as `<div>` (not `<h{n}>`) inside each entry card, with the content paragraph as `<p>`. Since each entry is semantically a `<li>` in a list, a screen-reader user can navigate the list fine; but there's no programmatic link between the author-name string and the content. Severity: **Minor** — Fix: wrap the author/date block in a `<header>` and mark the content with `aria-labelledby`, OR use the `<h3>` for the author's name consistent with how `AppointmentCard.tsx:57` uses `<h3>` for the title.

- **`src/components/dashboard/NextAppointments.tsx:70-113`** — `AppointmentRow` renders the row as a bare `<div>` (not an `<article>`) and the title is `<h3>` (`line 81`) with the time above it as `<div>`. Good that there's a heading; however the "no driver" state at lines 95-109 uses `<button type="button"…>` rendered as a link-styled inline element with only color + a tiny amber dot to distinguish it — see finding 1.4.1 below.

- **`src/components/info/ContactList.tsx:92-180`** — Contact rows use **nested interactive elements**: each row contains a `<button>` covering name + avatar (the toggle, lines 100-123) AND a separate `<a href="tel:">` (lines 124-134). This is legal in HTML but unusual. The *really* fragile bit is at line 103: `className="contents text-left outline-none"` on the button — `display: contents` strips the button from the layout tree. Screen readers traverse `<button>` + `<span>` + `<span>` correctly, but `contents` breaks focus-ring rendering on macOS Safari (known Safari bug with `display: contents` on focusable elements). Severity: **Major** — Fix: either restructure so the grid is inside the button, or give the button an explicit `display: block` focus state with a visible outline.

- **`src/components/info/EntitlementList.tsx:99-170`** — Each card has an invisible full-card edit button (`absolute inset-0`, line 110) **and** an inline "claim" button (line 155) with `pointer-events-auto` sitting above the invisible button. The claim button works; the invisible edit button has `aria-label={tCommon("edit")} — {entitlement.title}` which is great. But the card also has `<h4>` + `<p>` in a visual hierarchy that screen readers flatten into a button name. A screen reader in browse mode hears the full card body as the button's label, which is fine; in forms mode the visual hierarchy is lost. Severity: **Minor** — Fix: consider a dedicated pencil button instead of a whole-card overlay, consistent with LogFeed / AppointmentCard.

#### 1.4.10 Reflow (Level AA)

#### 1.4.11 Non-text contrast (Level AA — 3:1 for UI components)

## Cross-cutting observations

### 1. `--muted-foreground` is wrong at the token level

The app binds `--muted-foreground` to `ink-faint (#8f958b)` (`globals.css:39`). At 2.87:1 on paper this fails WCAG 1.4.3 AA **by design, globally**. Every "secondary" text use inherits the failure — dashboard metadata, activity-feed timestamps, loading states, inactive tab labels, calendar eyebrow text, empty-state descriptions. This is not 30 individual code bugs, it is **one token bug multiplied 30×**. The fix is a single-line change in `globals.css` to bind `--muted-foreground` to `ink-soft (#5a6158)` (5.97:1, PASS), plus a global find-replace of `text-ink-faint` → `text-ink-soft` *on text elements only* (keep `ink-faint` as a line/divider colour where 3:1 applies).

### 2. The `ring/50` + `input = divider-strong` pairing makes both borders and focus rings barely visible

Two tokens — `--ring` used at 50% alpha throughout button/input/select focus states, and `--input = --divider-strong` at 14% alpha — fall below WCAG 1.4.11's 3:1 threshold for non-text UI components. The combined effect: the boundary of every form field is at ~1.5:1 (near-invisible), and the focus indicator on every interactive control is at ~2.25:1 (faint). For a 60+ low-vision audience, this materially reduces operability. Both fixes are token-level changes in `globals.css` — not per-component rewrites.

### 3. The shadcn defaults are sized for desktop-power-users, not Sigga's audience

The `button.tsx` variant table has five sizes below 48 px (`xs`, `sm`, `default`, `lg`, `icon-*`). Every call site that chooses anything other than `touch` / `touch-icon` is a project-floor violation. Rather than audit every call site, the stricter fix is to **remove the sub-48 px variants** from `button.tsx` or deprecate them in the codebase (via Biome rule or code-review check). The plan's `docs/ux-patterns.md` should probably codify: "only `touch` and `touch-icon` are legal in user-facing screens; `sm` etc. are dev-admin-only". Same logic applies to `input.tsx:11` (default `h-8` = 32 px — every form explicitly overrides to `h-12`, which should be the default).

### 4. dnd-kit is accessibility-aware but the app doesn't opt in

Both DragOverlay usages (WeekGrid, EntitlementKanban) omit the `announcements` prop. This is a one-file-each fix and would turn a silent drag-and-drop into a narrated one ("Driver Helga lifted. Over tenth appointment: Læknaheimsókn. Dropped."). It's the single biggest a11y win available per line of code.

### 5. The heading hierarchy is broken on every page

Every top-level route starts at `<h2>`. The root cause seems to be that historically the sticky `<header>` + `<h1>Sigga</h1>` lived in a previous layout and got removed when the Sidebar + Header combination shipped — but no `<h1>` was added on the inner pages to replace it. One-file fix per page.
